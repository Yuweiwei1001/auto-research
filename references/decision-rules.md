# 决策规则

auto-research Skill 在 Phase 1（制定计划前）和 Phase 3（批次汇总后）
应用以下决策规则，确定下一步行动。

所有规则均为领域无关设计，使用 `research_domain.yaml` 中定义的阈值。

---

## 规则表

| # | 规则名 | 触发条件 | 动作 |
|---|--------|---------|------|
| 1 | 持续改进 | 主指标连续 2 个实验提升 ≥ `upgrade_threshold` | 当前方向继续微调，提出更精细的假设 |
| 2 | 停滞 | 连续 3 个实验提升 < `stagnation_threshold` | 建议换方向或引入全新思路 |
| 3 | 退化 | 主指标低于当前最佳 | 回滚到最佳版本，分析退化原因 |
| 4 | 达成目标 | 主指标 > `target_value` 且稳定 (std < `confidence_threshold`) | 报告成功，建议多设置验证 |
| 5 | 反复失败 | 连续 `max_consecutive_failures` 个实验报错且修复失败 | 暂停批次，请用户审核 |
| 6 | 深度验证升级 | 快速筛选主指标提升 ≥ `upgrade_threshold` | 同一假设升级为深度验证 |
| 7 | 完整运行升级 | 深度验证 std < `confidence_threshold` | 升级为完整运行 |
| 8 | 多种子验证 | 深度验证提升 ≥ `upgrade_threshold` 且稳定 | 同配置跑 3+ 个不同种子，报告 mean ± std |
| 9 | 变量扩展 | 当前方向在默认设置下有效 | 测试 `key_variables` 中的其他值 |
| 10 | 跨设置验证 | 当前方向在单一设置下有效 | 测试不同实验条件组合 |
| 11 | 批次预算耗尽 | 本批次实验数达到 `budget.max_experiments_per_batch` | 强制停止，输出中间报告 |
| 12 | 全局预算耗尽 | 累计实验数达到 `budget.max_total_experiments` | 暂停所有循环，请用户审核 |
| 13 | 时间预算超限 | 本批次运行时间超过 `budget.max_hours_per_batch` | 完成当前实验后停止 |
| 14 | 退化冷却 | 触发规则 3（退化）后 | 下一批次实验数减半，进入冷却期 |
| 15 | 早期停止 | 实验运行中主指标已低于当前最佳且剩余进度 > 30% | 建议子代理终止实验，标注 outcome=early_stop |

---

## 阈值配置

所有阈值从 `research_domain.yaml` 动态读取：

```yaml
decision_thresholds:
  upgrade_threshold: 0.01       # 提升 ≥ 1% 视为 improvement
  stagnation_threshold: 0.005   # 提升 < 0.5% 视为停滞
  neutral_threshold: 0.003      # 差异 < 0.3% 视为 neutral
  confidence_threshold: 0.01    # std < 1% 视为稳定
```

---

## 规则优先级

当多条规则同时触发时，按以下优先级执行：

1. **规则 12（全局预算耗尽）** > 所有其他规则 → 立即停止一切
2. **规则 11/13（批次预算/时间超限）** > 所有其他规则 → 完成当前实验后停止
3. **规则 5（反复失败）** > 所有其他规则 → 立即暂停
4. **规则 4（达成目标）** > 规则 1-3 → 切换到验证模式
5. **规则 6/7（升级验证）** > 规则 1-3 → 先升级验证再继续
6. **规则 8（多种子验证）** > 规则 1-3 → 先多种子验证再决定下一步
7. **规则 3（退化）** > 规则 1-2 → 回滚并分析（同时触发规则 14 冷却）
8. **规则 2（停滞）** > 规则 1 → 换方向
9. **规则 15（早期停止）** 可在任何实验运行中触发 → 终止当前实验

---

## 决策流程

```
循环启动
  │
  ├── 全局实验数 >= max_total？ → [规则12] 停止一切，请用户审核
  ├── 本批次实验数 >= max_per_batch？ → [规则11] 强制停止，输出中间报告
  ├── 本批次运行时间 >= max_hours？ → [规则13] 完成当前实验后停止
  │
实验完成
  │
  ├── 连续 N 个失败且修复失败？ → [规则5] 暂停，等用户
  │
  ├── 主指标 > target 且 std < confidence？ → [规则4] 切换到多设置验证
  │
  ├── 快速筛选提升 ≥ upgrade_threshold？ → [规则6] 升级为深度验证
  │   ├── 深度验证提升 ≥ threshold 且稳定？ → [规则8] 多种子验证
  │   │   └── 多种子 std < confidence？ → [规则7] 升级为完整运行
  │   │       └── 完整运行通过？ → [规则9/10] 扩展变量/设置
  │   └── 深度验证 std ≥ confidence？ → 继续快速筛选其他方向
  │
  ├── 主指标 < 当前最佳？ → [规则3] 回滚 + 分析 + [规则14] 冷却
  │
  ├── 连续 3 个 < stagnation_threshold 提升？ → [规则2] 换方向
  │
  └── 其他 → [规则1] 继续微调
```

---

## "换方向" 的具体含义

当触发规则 2（停滞）时，"换方向"意味着：

1. **回顾实验历史**：哪些方向试过但无效？哪些还没试过？
2. **回到创新来源**：重新检索论文和报告，寻找未探索的技术路线
3. **提出全新假设**：不基于当前方法微调，而是引入新机制
4. **可能的方法变更**：
   - 修改核心算法/模型架构
   - 引入新的损失函数/目标函数
   - 更换核心策略/范式
   - 参考完全不同的论文思路
   - 跨领域方法迁移
5. **在 method_lineage.yaml 中创建新方法版本**

## 批次决策报告中的决策部分

每个批次报告必须包含：

```markdown
## 决策

**触发的规则**: 规则 X — {规则名}
**依据**: {具体的 metrics 数据}
**下一步行动**:
- [ ] {具体行动 1}
- [ ] {具体行动 2}
- [ ] {具体行动 3}

**下一批次预期实验数**: N (快速筛选 X + 深度验证 Y)
**预计耗时**: ~X 小时
```

## 基线刷新规则

在以下情况下，先运行基线实验再继续创新实验：

1. **实验条件变更**：更换了关键变量值
2. **首次在新设置下实验**：任何 key_variable 值之前未测试过
3. **基线来源为论文声称**：需要本地实测验证

基线实验运行所有 `research_domain.yaml` 中定义的 baselines，
更新配置文件中的 metrics 值。

## 预算配置

预算参数定义在 `research_domain.yaml` 的 `budget` 字段中：

```yaml
budget:
  max_experiments_per_batch: 15    # 单批次最多实验数
  max_hours_per_batch: 8           # 单批次最长运行时间（小时）
  max_total_experiments: 200       # 全局实验总数上限
  max_consecutive_failures: 2      # 连续失败上限（触发规则5）
  cooldown_after_regression: 1     # 退化后冷却批次数（规则14）
  cooldown_experiment_cap: 5       # 冷却期间单批次实验数上限
```

### 预算追踪

主代理在每个批次中维护以下计数器：

- `batch_experiment_count`：本批次已执行实验数
- `batch_start_time`：本批次开始时间戳
- `total_experiment_count`：全局累计实验数（从 experiment_log.jsonl 统计）
- `cooldown_remaining`：剩余冷却批次数

每个实验开始前和完成后检查预算状态，超限则触发对应规则。
